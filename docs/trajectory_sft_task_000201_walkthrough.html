0 · 先写答案

这条轨迹训练的到底是什么？

答案：仍然是 decoder-only 语言模型的“预测下一个 token”。不同之处不是换了新 loss， 而是把 assistant 的中间推理、工具调用和最终回答都作为正确答案；system、user、工具定义和工具返回只作条件。
Lsample = − (1 / 416) · Σj∈A log Pθ(xj | x<j) A 是本样本 416 个监督 token 的位置集合；其余 3,048 个位置不直接计入交叉熵。

对这条数据，模型要学习三类行为：先分析需要查什么、再生成可执行的 Bash/SQL 工具调用、最后根据工具返回的 6639 给出“6,639 分钟”的答案。工具返回本身不是模型应该伪造的输出，因此不作为预测目标。

5 · 精确划定切口

Loss mask：3,464 个 token 中只给 416 个打分

条件 / observation：3,048（87.99%）
监督：416（12.01%）

每 1 个监督 token，约有 7.33 个非监督上下文 token。它们仍消耗前向计算和显存。

5.1 三种 mask 不能混为一谈

机制控制什么本样本中的作用
Causal mask一个位置能不能看未来 token生成 SQL 时绝不能读取未来的 6639
Padding / attention mask批处理补齐位置是否进入注意力不同长度样本组 batch 时屏蔽 pad
Loss mask / labels该位置是否直接计入交叉熵只给 assistant 推理、调用、答案打分

5.2 按消息角色划分

区域留在 input_idslabels直接参与 loss理由
tools schema是-100否可用动作空间
system是-100否规则条件
user是-100否问题条件
assistant reasoning是真实 token id是学习分析过程
assistant tool call是真实 token id是学习工具名、SQL 与参数
tool response是-100否环境事实，不让模型伪造
assistant final是真实 token id是学习最终回答
padding批处理时存在-100否仅用于对齐

5.3 一个缩微标签切片

下列词块只展示语义分区，不代表 Qwen tokenizer 的真实切词边界。

[system] 物流分析师 [user] 2026-06-19 [assistant/thinking] 检查表结构 [tool_call] SELECT SUM(...) [tool_response] 6639 [assistant] 6639 分钟 
input_ids = [  ...系统..., ...用户..., ...推理..., ...SQL..., ...6639..., ...答案... ]
labels    = [ -100, ...-100...,  token, ...token, -100, ...-100..., token, ...token ]
mask m_j  = [    0, ...   0...,     1, ...   1,    0, ...   0...,     1, ...   1 ]
最重要的手术级区别：labels[j] = -100 只表示“这个位置不作为直接预测题目”。 它不表示删掉 input_ids[j]，也不表示该 token 对后续答案没有影响。

6.1 teacher forcing 的准确含义

训练时整条轨迹已经在内存中，但模型在位置 t 的概率仍只条件于真实前缀 x<t。 “整条轨迹并行计算”是计算优化，不是让早期位置读取后文。

Pθ(x1:T) = ∏j=1…T Pθ(xj | x<j)

6.2 observation 不计 loss，却仍能影响梯度

直接项

6639 的 label 是 -100，所以没有“请模型预测 6639”这一项交叉熵。

间接路径

后续“6639 分钟”的预测会注意到 observation。其 loss 反向传播时，梯度取决于模型如何利用这段上下文。

因而更精确的说法是：tool response 没有直接 target loss，但它参与后续 supervised token 的条件计算。 “tool response 完全没有梯度作用”是错误表述。
7 · 像解数学题一样计算

从 logits 到单条轨迹的 masked cross-entropy

7.1 第一步：产生 logits

模型读入前缀 x<j，在词表的每个候选 token 上给出一个未归一化分数：

zj−1 = fθ(x<j) ∈ ℝ|V|

7.2 第二步：softmax 得到概率

Pθ(xj=v | x<j) = exp(zj−1,v) / Σu∈V exp(zj−1,u)

例如某个监督位置只有三个候选，logits 为 [2, 1, 0]，正确 token 是第一个：

p = e² / (e² + e¹ + e⁰) ≈ 0.665；　ℓ = −log(p) ≈ 0.408

7.3 第三步：乘 mask，只保留 416 道题

令 mj∈{0,1} 表示 token xj 是否受监督：

ℓj = −mj · log Pθ(xj | x<j)

因此本样本在标准非忽略 token 均值口径下：

Lsample = − 1/416 · Σj=2…3464 mj log Pθ(xj | x<j)
索引移位：位置 j−1 的 logits 预测位置 j 的真实 token。实现里常见 logits[..., :-1, :] 对齐 labels[..., 1:]；不要把“当前位置输入”误写成“预测它自己”。

7.4 batch 级归一化

Lbatch = − Σi,j mi,j log Pθ(xi,j|xi,<j) / Σi,j mi,j

上式是常见 token-level mean。分布式训练还需跨 data-parallel rank 聚合分子和有效 token 数；若具体 trainer 使用不同归一化口径，应以运行代码和日志为准。

8 · 缝合与恢复

反向传播、梯度与参数更新

8.1 反向传播

g = ∇θLbatch

416 个监督 token 提供直接交叉熵项。system、user 和 tool response 不产生自己的 target loss，但它们影响监督位置的 hidden state， 所以模型仍会学习“怎样利用这些条件作出后续动作”。

8.2 AdamW 的抽象更新

mt = β₁mt−1 + (1−β₁)gt
vt = β₂vt−1 + (1−β₂)gt²
θt+1 = θt − η · m̂t/(√v̂t+ε) − ηλθt

学习率 η、权重衰减 λ、梯度裁剪、混合精度和梯度累积会影响数值过程，但不改变“只在监督 token 上定义目标”的本质。

8.3 全参数与 LoRA

全参数 SFT

θ 中被允许训练的基础模型参数都接收更新。本项目正式链路采用 Megatron/MindSpeed 分布式全参数训练。

LoRA SFT

基础权重冻结，只更新低秩增量：W = W₀ + (α/r)BA。数据、mask 和 cross-entropy 不变。

一次 optimizer step 不等于一次完整轨迹。如果使用 micro-batch、gradient accumulation、sequence packing 或数据并行， 必须根据训练配置说明“一步”实际聚合了多少样本和多少监督 token。
9 · 手术记录

从一行 JSON 到一次更新：九步闭环

身份与切分核验

确认 task_000201 属于 train，且同题变体没有泄漏到 validation/test。

输入：源文件、行号、task_id、plan metadata
输出：可进入 train 的唯一身份

正确性与结构核验

确认 SQL 结果证据、4 次调用/返回闭环、最终回答存在、参数 JSON 可解析。

输入：OpenAI-style messages
输出：sql_result_verified 合格轨迹

格式转换

推理合入 <think>，tool_calls 拆为 tool_call，工具输出变为 tool_response，并补工具 schema。

输入：OpenAI-style record
输出：ms-swift agent row

Qwen 模板序列化

把工具定义、角色边界和消息内容编码为 Qwen3.5 真正看到的单一字符串。

输入：tools + messages
输出：带特殊标记的 prompt 序列

真实 tokenizer 编码

生成长度为 3,464 的 input_ids；长度边界按 token 判定，不按字符判定。

输入：序列化字符串
输出：x₁…x₃₄₆₄

构造 labels

assistant reasoning/tool_call/final 保留 token id，其余位置设为 -100；得到 416 个监督位置。

输入：消息边界 + input_ids
输出：labels 与 loss mask

因果前向传播

所有位置并行算 logits，但每个位置只能注意左侧；生成 SUM SQL 时不可见未来 6639。

输入：input_ids + causal/padding mask
输出：每个位置的词表 logits

Masked cross-entropy

对齐 shifted logits/labels，只汇总 416 个非 -100 目标；tool response 没有直接 loss。

输入：logits + labels
输出：标量 loss

反向传播与 optimizer step

计算梯度、跨设备聚合、裁剪并更新全参数或 LoRA 参数；保存可复现日志与检查点。

输入：标量 loss
输出：更新后的 θ

10.1 这条样本能证明什么

  • 能示范完整的工具轨迹监督结构
  • 能证明当前模板的 tool call 被监督
  • 能证明 tool response 被保留但不计直接 loss
  • 能给出真实长度与监督密度

10.2 这条样本不能单独证明什么

  • 不能证明 2,028 条全部答案正确
  • 不能证明训练后一定会在线调用成功
  • 不能证明更长轨迹的显存一定足够
  • 不能用单样本训练 loss 代替 held-out 评测
最后的安全线：轨迹“格式正确”只说明可训练；只有工具结果可验证、task 级切分无泄漏、在线闭环评测通过，才说明它对 agent 能力真正有价值。

D. 60 秒讲解词

这条数据先被转换成 Qwen 能识别的完整工具交互序列，共 3,464 个 token。训练时模型能读取 system、用户问题和工具返回， 但只有 assistant 的推理、工具调用和最终回答这 416 个 token 参与交叉熵。模型在生成 SUM SQL 时受因果 mask 限制，看不到未来的 6639； 生成最终答案时，6639 已经是历史 observation，所以模型学习的是“根据真实工具结果回答”。最后，416 个监督位置的负对数概率求平均并反向传播， 更新全参数或 LoRA 参数。这就是轨迹 SFT：loss 仍是 next-token cross-entropy，但监督对象从一个答案扩展成了完整的行动过程。
