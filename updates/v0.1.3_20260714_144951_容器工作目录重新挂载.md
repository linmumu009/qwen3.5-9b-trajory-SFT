# 将训练容器工作区重新挂载到统一项目目录

## 一、更新概述

安全重建 `qwen3.5-9b-msswift-trajory-sft` 容器，将容器内 `/workspace/sft` 的宿主机来源从旧目录切换为 `/data3/llin/trajory_sft`。旧容器未删除，而是停止并重命名保留，便于快速回滚。

## 二、操作前检查

- 目标容器仅运行空闲 bash，没有训练或数据处理任务。
- 旧挂载目录 `/data3/llin/sft` 约 1.9 TiB。
- 新工作目录 `/data3/llin/trajory_sft` 为空。
- 本轮没有复制或移动旧目录中的 1.9 TiB 数据。

## 三、具体改动

### 1. 保留旧容器

停止原目标容器，并重命名为：

```text
qwen3.5-9b-msswift-trajory-sft-backup-v0.1.2
```

该备份容器当前为 `exited` 状态，保留旧挂载配置和容器可写层。

### 2. 使用新挂载重建容器

使用原镜像和原容器名称重新创建容器：

```text
容器: qwen3.5-9b-msswift-trajory-sft
镜像: qwen3.5-9b-msswift-trajory-sft:latest
容器 ID: 8648f4a3afc4eeea7055c421e336209608321e6aa981064fbbb31a57f86a98da
```

工作区挂载调整为：

```text
/data3/llin/trajory_sft -> /workspace/sft (读写)
```

以下配置保持不变：

- Ascend runtime
- `host` 网络模式
- `host` IPC 模式
- privileged 模式
- `label=disable` 安全选项
- 模型目录只读挂载
- Ascend 驱动、add-ons 和系统工具挂载
- 原镜像 Entrypoint、Cmd、环境变量和工作目录

## 四、验证结果

- 新容器状态：`running`
- 源容器 `qwen3.6-27b-msswift-sft` 状态：`running`
- 回滚备份容器状态：`exited`
- Docker inspect 确认 `/workspace/sft` 来源为 `/data3/llin/trajory_sft`
- 在容器内创建临时文件后，宿主机新目录中可立即读取，证明读写挂载生效
- 临时挂载测试文件已删除，新目录恢复为空
- Python：3.11.15
- `npu-smi info` 执行成功，Ascend 910 设备 0–15 均可见且健康
- 验证期间未启动训练任务，也没有 NPU 计算进程

## 五、兼容性与影响

- 新容器的 `/workspace/sft` 当前为空，不再显示旧 `/data3/llin/sft` 中的代码、数据和输出。
- 旧目录约 1.9 TiB 的内容保持原样，没有复制、移动或删除。
- 源容器仍继续使用旧目录，不受本轮重新挂载影响。
- 后续所有 Qwen3.5-9B 轨迹 SFT 文件应写入 `/data3/llin/trajory_sft`，并在新容器内通过 `/workspace/sft` 访问。

## 六、备份与回滚

如需恢复旧挂载，可停止并删除当前新容器，将备份容器改回原名后启动：

```bash
docker stop qwen3.5-9b-msswift-trajory-sft
docker rm qwen3.5-9b-msswift-trajory-sft
docker rename qwen3.5-9b-msswift-trajory-sft-backup-v0.1.2 qwen3.5-9b-msswift-trajory-sft
docker start qwen3.5-9b-msswift-trajory-sft
```

删除容器不会删除宿主机 `/data3/llin/trajory_sft` 中的数据，但执行回滚前仍应确认其中的重要文件已妥善保存。
