# v0.2.0 Qwen3.5 轨迹 SFT 训练镜像构建

- 版本：v0.2.0
- 日期：2026-07-14
- 更新类型：训练环境建设
- 状态：已完成

## 更新概述

以官方 Ascend A3 镜像 `quay.io/ascend/ms-swift:v4.3.0-A3-py311-CANN9.0.0-ubuntu22.04` 为基座，构建了面向 Atlas 900 A3（`ascend910_9391`）、Qwen3.5-9B 和 ms-swift 轨迹 SFT 的派生训练镜像。补齐 Qwen3.5 线性注意力在 NPU 上需要的 MindSpeed Triton 快速算子后，用本地 Qwen3.5-9B 真实权重完成了前向和 LoRA 训练步验证。

最终删除旧的同名容器，以新镜像重建 `qwen3.5-9b-msswift-trajory-sft`，保留原容器名、Ascend 运行时参数和工作目录挂载。

## 具体改动

1. 新增 `docker/Dockerfile.a3-qwen35`：
   - 继承官方 A3 / CANN 9.0.0 镜像。
   - 锁定 ms-swift、MindSpeed 和 Megatron-LM 的 GitCode 仓库及提交号。
   - 仅使用阿里云 PyPI 镜像安装新增 Python 包。
   - 在构建期检查关键版本和 Qwen3.5 快速算子源文件。
2. 新增 `scripts/verify_qwen35_npu.py`，校验依赖版本、MindSpeed 算子模块、SoC 型号、16 卡可见性和最小 NPU 张量运算。
3. 新增 `scripts/smoke_qwen35_model.py`，对本地 Qwen3.5-9B 真实权重执行补丁断言、前向计算以及可选的 LoRA 训练步。
4. 将上述文件同步到服务器 `/data3/llin/trajory_sft`，并构建镜像 `qwen3.5-9b-msswift-trajory-sft:v0.2.0`。
5. 删除旧的 `qwen3.5-9b-msswift-trajory-sft` 容器，用 v0.2.0 镜像按原配置重建。

## 构建调整记录

- 原始 Gitee Megatron-LM 镜像可查询引用，但匿名克隆会要求凭据，因此切换到 GitCode 公开镜像。
- ms-swift 4.3.0 使用旧的 `mindspeed.lite.ops.triton` 导入路径，而 MindSpeed 0.16.0 已迁移到 `mindspeed.ops.triton`；升级至 ms-swift 4.3.2 后路径一致。
- Docker 构建阶段不挂载宿主机 Ascend 驱动，因此构建内只做版本和源文件检查；模块导入、NPU 张量和模型训练验证均在启动后容器中完成。
- MindSpeed 0.16.0 元数据声明 `numpy<=1.26.0`，而官方基础镜像中的 `triton-ascend 3.2.1` 严格要求 `numpy==1.26.4`。本镜像保留 1.26.4，并通过了实际 Qwen3.5 前向和反向训练验证。

## 已知边界与遗留问题

- 本镜像定位为 **ms-swift 轨迹 SFT 训练环境**。官方基础镜像预装的 vLLM 0.18 / vllm-ascend 0.18 与当前 Qwen3.5 训练栈在 Transformers、torch-npu 和 triton-ascend 版本上存在冲突；vLLM 推理、rollout 和部署未在此镜像中验证，如后续需要应建立独立镜像。
- 极短的 8 token smoke 序列会触发 MindSpeed 的形状启发式警告，但前向、反向和优化器更新均完成；正式训练仍需用实际序列长度做小批次验收。
- 本轮未修改、转换或导入轨迹数据；数据协议和轨迹 mask 规则将在下一阶段单独验证。

## 回滚方式

官方 A3 基础镜像仍保留在服务器。如需回滚，停止并删除当前同名容器，用 `quay.io/ascend/ms-swift:v4.3.0-A3-py311-CANN9.0.0-ubuntu22.04` 按本报告记录的 Ascend 运行参数和原挂载重建即可。回滚只影响容器环境，不会删除 `/data3/llin/trajory_sft` 或 `/data3/llin/base_model` 中的宿主机数据。
