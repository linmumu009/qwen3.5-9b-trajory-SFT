## 更新概述

将 `/data/renjunxiang/coding/huawei_train/datasets/sandboxes/sft_data` 中的未筛选 OpenAI 轨迹数据完整复制到项目统一目录，对 36,000 条数据完成格式、工具闭环、manifest/verdict 对齐和 ms-swift 兼容性审计。

原始数据虽是 OpenAI messages 形式，但不能直接用 ms-swift 4.3.2 做轨迹 SFT：ms-swift 的通用预处理会删除 `assistant.reasoning_content` 和 `assistant.tool_calls` 这两个额外字段。本轮新增适配器，将 reasoning 合并为 `<think>...</think>`，将 OpenAI tool call 转换为 ms-swift `tool_call`，将 tool 返回转换为 `tool_response`，并用 `default+ignore_empty_think` 使中间思考、工具调用和最终回答参与训练，tool response 不参与 loss。

最终抽取 8 条 verdict 为 `correct`、工具往返完整且长度较短的轨迹，用 Qwen3.5-9B 在 Atlas 900 A3 NPU 上完成一次真实 ms-swift LoRA optimizer step。

## 现有判分脚本评估

`judge_trajectory.py` 的输入不是 OpenAI JSONL，而是包含 `output` 原始事件轨迹路径的 manifest。它需要：

1. 从原始事件中提取最终 assistant 答案、工具命令和结束状态。
2. 访问 `/data/renjunxiang/pi/sandbox/sft/<version>/logistics.sqlite`。
3. 用 manifest 的 `gold_answer/verification_sql/expected_tables` 进行规则判分。

因此它不能直接指向 `sft_data/*.jsonl`。服务器实际已有一轮 fixed verdict，但 verdict 与 SFT JSONL 分开存放，所以需要追溯关联而不是对已转换文件重新盲判。

原判分规则也不等于完全真值：

- DWH 报告型任务主要检查 complete、答案长度和 expected tables，不核验报告的所有事实和数值。
- KB 任务主要检查 gold 数值是否出现和答案长度，语义完整性校验较弱。
- 数值/table 任务会对比 gold SQL 与 agent SQL 结果，可靠性更高，但 SQL 提取仍依赖规则正则。
- 本轮发现 verdict=`correct` 中仍存在未返回 tool call，证明正式 SFT 必须叠加轨迹结构检查。

## OpenAI 到 ms-swift 轨迹转换

`scripts/prepare_trajectory_sft.py` 执行：

1. 使用 `_source` 或“原文件名排序 + user instruction”对齐 task_id。
2. 连接 fixed verdict，默认只保留 `correct`。
3. 将 `reasoning_content` 包装为 Qwen3.5 `<think>...</think>`。
4. 将每个 OpenAI `assistant.tool_calls[]` 转为 ms-swift `role=tool_call`。
5. 将 OpenAI `role=tool` 转为 ms-swift `role=tool_response`。
6. 确认 tool call ID 唯一、tool response ID 存在、每个调用有且只有一个返回。
7. 训练 JSONL 仅保留 ms-swift 标准字段；来源文件、行号、task_id、verdict 和转换统计写入独立 sidecar。

`scripts/verify_trajectory_encoding.py` 在 Qwen3.5 模板编码后验收：

- `<think>` 存在。
- `<tool_call>` 在 input 和 labels 中存在，工具调用参与 loss。
- `<tool_response>` 在 input 中存在，但不出现在 supervised labels 中。
- 8 条 smoke 数据编码长度为 1,405–2,458 tokens，监督 token 为 306–834。

## 真实 ms-swift 训练 smoke

参数要点：

- 模型：`/models/Qwen3.5-9B`
- 卡：NPU 0
- 训练方式：LoRA，`target_modules=all-linear`，`rank=4`，`alpha=8`
- `max_length=2560`，8 条数据均不截断
- `loss_scale=default+ignore_empty_think`
- batch size 1，gradient accumulation 1，gradient checkpointing 开启

## 正式筛选建议

### 硬门槛

- 只从 verdict=`correct` 开始。
- 通过身份对齐、调用/返回闭环、最终 assistant、规范工具参数检查。
- 使用 Qwen3.5 tokenizer 计算真实 token 长度，按训练长度预算分桶，不用字符数代替 token 数。
- 不在轨迹中间盲截断；超长样本应整条排除或单独用长序列配置训练。
- train/validation/test 必须按 task_id 分组切分，防止同一任务的不同模型/沙箱版本跨集泄漏。

### 去重与质量排序

- 第一阶段每个 `(version, task_id)` 最多保留 1 条轨迹。
- 优先选择 SQL/工具无错误、最终答案完整、调用次数较少且路径直接的轨迹，不应单纯以最短为优。
- 可先以 P95 工具调用数 58 为审计线，对高于该值的轨迹检查是否存在重复查询或无效探索。
- 报告型和 KB 型 `correct` 需优先复判；数值/table 型且 agent SQL 与 gold SQL 结果一致的样本可先进入高置信度集。

### 非 SFT 样本

- `partial`：不作 SFT 正样本；如同 task_id 存在高质量 correct，可组成 DPO 偏好对候选。
- `incorrect`：不作 SFT；可作 DPO rejected 候选，但需避免将安全不可执行内容直接纳入。
- `incomplete`：直接丢弃。
