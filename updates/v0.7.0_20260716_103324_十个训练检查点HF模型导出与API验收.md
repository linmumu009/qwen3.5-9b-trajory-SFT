## 导出方案

固定版本文档要求使用分布式 `megatron export --mcore_model ... --to_hf true` 完成 MCore 到 Hugging Face 的转换。本轮保持训练权重的 `TP4 / PP2` 切分，使用 8 个进程构成一个完整模型副本；训练时的第二个数据并行副本不参与导出。

导出脚本具有以下保护：

1. 只读取已成功结束的训练运行，不修改或覆盖源检查点。
2. 校验检查点目录与迭代标记一致。
3. 先写入隐藏临时目录，只有 HF 文件结构检查通过后才原子改名为正式目录。
4. 发现已存在但不完整的目标目录时拒绝覆盖；已完整导出的目录可安全跳过，支持续跑。
5. 输出采用 BF16 safetensors，不加载、不导出 optimizer 与 RNG。

ms-swift 对已接入模型允许关闭额外转换精度对照，因此本轮设置 `test_convert_precision=false`；随后使用 safetensors 全量头部检查、Transformers 元数据加载和真实生成 API 请求验证导出产物。

## API 启动方式

容器使用 host 网络，因此服务启动后可直接访问物理机对应端口。以下示例启动 step 150：

```bash
docker exec -it \
  -e MODEL_DIR=/workspace/sft/exports/qwen35_9b_megatron_16k_2028_150steps_hf/checkpoint-150-hf \
  -e SERVED_MODEL_NAME=qwen3.5-9b-trajory-sft-step150 \
  -e ASCEND_RT_VISIBLE_DEVICES=0 \
  -e PORT=8000 \
  qwen3.5-9b-msswift-trajory-sft \
  bash /workspace/sft/scripts/serve_exported_qwen35.sh
```

调用示例：

```bash
curl http://127.0.0.1:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "qwen3.5-9b-trajory-sft-step150",
    "messages": [{"role": "user", "content": "你好"}],
    "max_tokens": 256,
    "temperature": 0
  }'
```

替换 `MODEL_DIR` 与 `SERVED_MODEL_NAME` 即可部署其余 9 个检查点。启动脚本显式设置 `--load_args false`，确保服务使用指定 HF 导出目录，不依赖训练检查点中的原始模型路径。

## 回滚与安全边界

- 10 份原始 MCore 检查点保持不变，可用于断点续训或重新导出。
- HF 模型统一写入独立 `exports/` 目录，没有覆盖训练运行目录。
- 导出脚本拒绝覆盖不完整目标，失败时保留带 PID 的隐藏临时目录供排障。
- API 验收仅临时使用单张 NPU，结束后已关闭服务。
- 模型权重、服务器日志和大型验收产物不提交 GitHub；仓库只提交脚本与版本说明。
