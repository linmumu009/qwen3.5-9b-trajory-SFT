# Next-token 预测与目标侧 mask 说明

## 一、更新概述

继续修订 `docs/trajectory_sft_process.md`，回答轨迹 SFT 中最具体的问题：

> 到底是哪个位置预测哪个位置？哪个目标被 mask？tool response 被 mask 后，紧接着的 assistant 是否还会训练？

本轮新增逐 token 的 shift 定义、target-side mask 公式、消息区段级预测链、`task_000201` 十段真实对照和最小 token 示例。

## 二、具体改动

### 1. 明确 next-token 的索引关系

新增定义：

$$
z_{j-1}=f_\theta(x_{\le j-1})
$$

位置 `j-1` 的 logits 预测位置 `j` 的真实 token。训练实现概念上对齐：

```python
shift_logits = logits[:-1]
shift_labels = labels[1:]
```

因此：

```text
logits[j-1] 预测 input_ids[j]
labels[j] 决定这次预测是否计入 loss
```

### 2. 强调 mask 在目标侧

新增逐位置 loss：

$$
\ell_j=
-\mathbf{1}[y_j\neq -100]
\log P_\theta(x_j\mid x_{<j})
$$

这意味着：

- 目标 token 属于 assistant reasoning/tool call/final answer → `labels[j]` 是真实 token id → 计入 loss；
- 目标 token 属于 tools/system/user/tool response/padding → `labels[j]=-100` → 不计入 loss；
- 前一个 token 属于什么 role，不会自动决定下一个 target 是否被 mask；
- tool response 末尾之后的 assistant target 仍然受监督，因为 mask 看的是 assistant 目标位置。

### 3. 新增区段级预测链

新增下列语义链路：

```text
TOOLS/SYSTEM/USER
  → ASSISTANT THINK       真实 labels，算 loss
  → TOOL CALL             真实 labels，算 loss
  → TOOL RESPONSE         labels=-100，不算 loss
  → NEXT ASSISTANT/FINAL  真实 labels，算 loss
```

同时明确“tool call 预测 tool response”只表示标准语言模型前向会产生相应 next-token logits；真实训练用 `-100` 忽略这些目标，真实推理中的 tool response 由外部执行器提供。

### 4. `task_000201` 十段逐项对照

按被预测的目标区段依次标明：

1. tools/system/user：`-100`；
2. 第 1 轮数据库定位 assistant 推理与调用：监督；
3. 第 1 轮目录 observation：`-100`；
4. 第 2 轮表枚举 assistant 推理与调用：监督；
5. 第 2 轮表列表 observation：`-100`；
6. schema 推理与 `.schema` 调用：监督；
7. 含 `event_time/dwell_minutes` 的 schema observation：`-100`；
8. 求和推理与 `SELECT SUM(...)` 调用：监督；
9. SQL observation `6639`：`-100`；
10. 最终 assistant“总和为 6639 分钟”：监督。

由此明确：模型不学习凭空生成数据库结果 `6639`，而是学习在过去上下文已经含 `6639` 时生成正确后续回答。

### 5. 新增最小 shift 示例与误解修正

新增五个语义 token 的对齐例子，逐行写出 `logits[0]→input_ids[1]` 等关系，并在“常见误解”中增加：

- tool response 被 mask 不等于不产生 logits；
- 前一个 token 属于 tool response 不等于后续 assistant target 也被 mask。
