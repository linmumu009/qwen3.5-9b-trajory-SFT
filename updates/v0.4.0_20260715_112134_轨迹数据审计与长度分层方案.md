# v0.4.0 轨迹数据审计与长度分层方案

- 版本：v0.4.0
- 日期：2026-07-15
- 更新类型：数据质量审计、训练长度基准、数据规划
- 状态：审计与方案已完成；正式训练集仍待复判和路径归一化

## 技术摘要

本轮使用 Qwen3.5-9B 的真实 ms-swift 模板，对 36,000 条原始 OpenAI 轨迹重新执行格式门槛、工具参数校验、token 统计、loss mask 统计、内容风险审计、任务级去重和防泄漏切分，并在 Atlas 900 A3 64GB NPU 上实测 4K、8K、16K、32K 的 LoRA 训练显存。

结论如下：

1. 严格格式门槛后有 14,333 条候选，不应直接全量训练。候选中位长度 25,415 token，5,521 条超过 32K，却占全部输入 token 的 69.6%。
2. 全部候选共 403,934,946 输入 token，只有 75,084,199 token 参与 loss，加权监督占比仅 18.59%。工具返回文本的字符占比中位数为 85.86%，长轨迹的算力利用率是核心问题之一。
3. 原始 OpenAI 文件虽然有 tool call/tool response，但 14,333 条原始记录均没有顶层 `tools` schema，原 system prompt 也没有工具契约。转换器现已显式写入 bash/read/write/edit 的 JSON schema，并通过 Qwen3.5 模板注入与 loss mask 验收。
4. 现有 judge 的 `correct` 证据强度不一致。只有 182 条候选明确记录 `agent_sql_ok=true`；其余 14,151 条主要依赖报告长度、表命中、关键数字或其他规则，必须分层复判。
5. 元数据规划得到 4,259 条复判池，共 1,287 个唯一 task_id、69,242,138 输入 token。它是“可审查候选计划”，不是正式训练集；其中 4,179 条仍需复判，725 条还需路径归一化。
6. 单卡 LoRA rank 16 实测：4K 为 23.45 GiB，8K 为 25.80 GiB，16K 为 34.73 GiB，32K 为 59.95 GiB。32K 虽完成一步，但只剩约 4 GiB 物理余量，不适合作为生产配置。
7. 当前镜像的 32K 多卡路径尚未闭环：SP=2 需要额外 `flash-linear-attention` 内核；DeepSpeed 未安装；修正层类名后的 FSDP2 在真实前向仍因 59.32 GiB 已预留而 OOM。因此当前生产上限应设为 16K，32K 仅保留复判/实验池。

## 关键发现

### 1. 严格候选门槛

36,000 条原始记录中的 15,072 条被现有 verdict 标为 `correct`。严格门槛继续要求：

- user/assistant/tool 角色和消息结构可转换；
- tool call 与 tool response 数量闭环；
- 工具名只能为 bash/read/write/edit；
- 参数键、必填项、值类型和整数边界符合工具 schema；
- 存在可监督的 assistant 输出和最终回答。

最终得到 14,333 条候选：

| 项目 | 数量 |
|---|---:|
| 原始记录 | 36,000 |
| verdict=`correct` | 15,072 |
| 严格候选 | 14,333 |
| 转换错误排除 | 142 |
| 工具闭环失败排除 | 123 |
| 非规范工具/参数排除 | 474 |
| 唯一 task_id | 1,408 |
| 唯一 `(version, task_id)` | 12,485 |

统计使用 `/models/Qwen3.5-9B`、Qwen3.5 agent template 和 `default+ignore_empty_think` loss scale，顶层工具 schema 已计入输入。

| token 桶 | 条数 | 条数占比 | 输入 token | token 占比 |
|---|---:|---:|---:|---:|
| ≤2K | 18 | 0.13% | 31,620 | 0.01% |
| 2K–4K | 1,132 | 7.90% | 4,077,738 | 1.01% |
| 4K–8K | 2,403 | 16.77% | 14,221,679 | 3.52% |
| 8K–16K | 2,142 | 14.94% | 25,628,897 | 6.34% |
| 16K–32K | 3,117 | 21.75% | 78,997,987 | 19.56% |
| 32K–64K | 4,689 | 32.71% | 215,248,779 | 53.29% |
| 64K–131K | 824 | 5.75% | 64,533,505 | 15.98% |
| 131K–262K | 8 | 0.06% | 1,194,741 | 0.30% |

汇总分位数：最短 1,344，P50 25,415，P90 57,539，P95 67,316，P99 89,499，最长 188,867 token。Qwen3.5-9B 配置的理论上限是 262,144 token，但理论上限不等于训练显存可承受长度。

### 3. 监督密度与工具输出膨胀

- 输入 token：403,934,946
- 监督 token：75,084,199
- 加权监督占比：18.5882%
- 单条监督占比：P01 3.61%，P05 5.80%，P50 17.09%，P95 36.78%
- 监督占比低于 10%：2,543 条
- 工具返回字符占比：P50 85.86%，P95 95.66%，P99 97.28%
- 超过 4,096 字符的 tool response 消息：40,092 个
- 超过 16,384 字符的 tool response 消息：5,787 个
- 单个 tool response 最大：51,330 字符

工具返回需要进入上下文以保持轨迹因果关系，但默认不参与 loss。不能只按“条数”估算训练成本，也不能对所有工具结果做固定字符截断；应按任务语义保留形成结论所需的证据，并在压缩后重新验证最终答案和工具闭环。

### 4. 正确性证据并不等价

| 证据层级 | 条数 | 输入 token P50 | 说明 |
|---|---:|---:|---|
| `sql_result_verified` | 182 | 31,376 | agent SQL 结果与 gold 结果一致，证据最强 |
| `judge_rule_only` | 3,927 | 42,031 | 非报告、非 KB 的现有规则结论，仍需复核 |
| `report_rule_only` | 2,079 | 41,740 | 主要依赖完成状态、字数和表命中 |
| `kb_rule_only` | 8,145 | 10,819 | 主要依赖关键数字和答案长度 |

仅 1.27% 严格候选具有明确 SQL 结果一致证据。182 条强证据样本可以作为回归测试和高优先级种子，但数量不足以独立承担正式 SFT。

复判建议：

- DWH numeric/table：重放 agent SQL，与 verification SQL 结果比较，并验证最终回答数值和口径；
- KB：检查答案是否由指定文档支持，不再只做数字出现和长度判断；
- report：至少检查数据来源、结论与工具结果一致、无编造因果；可采用规则 + 独立 judge 的双重判定；
- hybrid：同时满足数据库结果、知识依据和最终综合结论；
- `partial/incorrect/incomplete` 不进入正向 SFT；partial/incorrect 可在构造可靠 chosen/rejected 对后用于 DPO。

### 5. 内容质量与安全

| 信号 | 记录数 | 处理建议 |
|---|---:|---|
| 正则命中工具错误文本 | 7,270 | 区分“报错后恢复”与“最终未恢复”；前者可保留少量，后者排除 |
| 任意重复 tool call | 905 | 作为排序惩罚，不一刀切 |
| 连续完全重复 tool call | 45 | 优先排除或压缩 |
| 非可移植绝对路径 | 3,176 | 在 prompt、调用和返回中一致替换为沙箱占位路径 |
| 网络命令 | 38 | 默认隔离，确认任务确实需要网络后再放行 |
| 破坏性命令 | 1 | 隔离，不进入 SFT |
| 高置信凭据模式 | 0 | 本轮规则未发现；仍需在物化后再次扫描 |

“工具错误”是启发式正则信号，不能直接等价为坏轨迹。轨迹 SFT 可以学习错误恢复，但应限制恢复型样本比例，并要求错误后出现有效修正和正确最终答案。

## 数据规划

### 1. 规划门槛

本轮规划按以下顺序执行：

1. 超过 32K 的样本进入隔离池；
2. 监督 token 占比低于 10%、最终回答少于 50 字、网络/破坏性命令或凭据风险样本不进入规划池；
3. 每个 `(version, task_id)` 只保留排序最优的一条；
4. 每个 task_id、每个长度阶段最多保留 3 条，优先保留不同来源模型；
5. 先按 task_id 稳定哈希切分，再物化 train/validation/test，防止同题不同版本泄漏。

规划共保留 4,259 条、1,287 个 task_id、69,242,138 输入 token 和 16,702,127 监督 token，加权监督占比提高到 24.12%。它只占原候选 17.14% 的输入 token，显著降低后续复判和训练成本。

### 2. 长度阶段

| 阶段 | token 范围 | 条数 | 唯一 task | 输入 token | 强证据 | 待复判 | 路径待归一化 |
|---|---:|---:|---:|---:|---:|---:|---:|
| `core_8k` | ≤8,192 | 893 | 397 | 3,742,018 | 10 | 883 | 37 |
| `extension_16k` | 8,193–16,384 | 1,385 | 654 | 15,916,947 | 22 | 1,363 | 290 |
| `long_32k_review` | 16,385–32,768 | 1,981 | 1,062 | 49,583,173 | 48 | 1,933 | 398 |

由于同一个 task_id 可以在不同长度阶段出现，各阶段唯一 task 数不能直接相加。

### 3. 防泄漏切分

目标比例是 task 级 85%/7.5%/7.5%，实际选中记录如下：

| split | 记录 | 唯一 task_id | 输入 token | 监督 token |
|---|---:|---:|---:|---:|
| train | 3,716 | 1,113 | 59,907,254 | 14,333,765 |
| validation | 315 | 95 | 5,277,793 | 1,349,895 |
| test | 228 | 79 | 4,057,091 | 1,018,467 |

记录比例与目标比例不完全相同，是因为同一 task_id 在不同版本/阶段拥有不同记录数；但同一 task_id 不会跨 split。

### 4. 来源和任务类型

规划池来源：qwen3.6-27b 3,811、qwen3.7-max 350、deepseek-v4-pro 74、glm-5.2 24。来源不应通过重复小来源样本强行做成均匀分布；更合理的做法是保持 task 上限、做来源分层评测，并补采新的非 Qwen 独立任务。

规划池类型：KB 2,589、DWH 736、Hybrid 934。但当前可安全训练的 ≤16K 两阶段合计 2,278 条，其中 KB 1,943、DWH 315、Hybrid 仅 20。也就是说，16K 首轮适合验证训练流程和 KB/DWH 基础能力，不足以代表完整 Hybrid 能力。

不建议重复上采样仅有的 20 条短 Hybrid。优先方案是把已复判正确的长 Hybrid 做语义压缩，或重新采集更短、更高监督密度的 Hybrid 轨迹。中期数据目标可设为 KB/DWH/Hybrid 约 50%/25%/25%，但必须通过新增唯一任务实现，而不是机械复制。

## 显存与长序列实测

### 1. 基准配置

- 模型：Qwen3.5-9B，本地权重
- 数据：真实完整轨迹，含顶层工具 schema
- 精度：BF16
- 微批：每卡 1
- LoRA：rank 16，alpha 32，`all-linear`
- gradient checkpointing：开启；FSDP2 使用其 activation checkpointing
- optimizer step：1
- 截断策略：`delete`
- loss scale：`default+ignore_empty_think`

### 2. 单卡结果

| 实际 token | max_length | 峰值显存 | 步耗时 | 结果 |
|---:|---:|---:|---:|---|
| 4,096 | 4,096 | 23.45 GiB | 38.21 s | 通过 |
| 8,192 | 8,192 | 25.80 GiB | 39.73 s | 通过 |
| 16,386 | 16,512 | 34.73 GiB | 45.68 s | 通过 |
| 32,766 | 32,768 | 59.95 GiB | 60.77 s | 通过但无安全余量 |

32K 的一步训练成功不代表可生产使用。64GB 物理 HBM 还要承受运行时预留、碎片、样本形状波动和评估；59.95 GiB 已接近可用容量。

### 3. 多卡长序列路径

- 普通 16 卡数据并行不会把单条序列拆到 16 张卡，只会让每张卡各跑一条序列；因此它提高吞吐，但不能解决单条 32K 的激活显存。
- SP=2 初始化后，在 Qwen3.5 线性注意力分支报错：需要额外 `flash-linear-attention` 内核。当前单卡 MindSpeed 快路径不能代替该序列并行分支。
- 当前镜像未安装 DeepSpeed，不能直接使用 `--deepspeed zero3`。
- ms-swift 内置 FSDP2 预设引用了旧层类名 `Qwen3_5TextDecoderLayer`。新增 `configs/fsdp2_qwen35.json` 后可正确包装当前 `Qwen3_5DecoderLayer` 并进入真实前向，但 32K 仍 OOM：运行时已预留 59.32 GiB，再申请 420 MiB 失败。

因此本镜像当前的生产上限设为 16K。32K 解锁条件是以下任一方案完成独立 20-step 验证：

1. 重新构建带可用 NPU `flash-linear-attention` 序列并行内核的镜像；
2. 在中国大陆可达源上构建并验证 DeepSpeed ZeRO-3；
3. 修复 FSDP2 内存预留/碎片并证明 32K 峰值低于约 52 GiB；
4. 将长 Hybrid 轨迹做语义压缩到 ≤16K，并重新判定答案与过程正确。

## 推荐训练方案

### 1. 当前可执行版本

正式开训前先完成 `core_8k` 和 `extension_16k` 的复判、路径归一化及物化。训练参数从以下保守配置开始：

- `--max_length 16384`
- `--truncation_strategy delete`
- `--per_device_train_batch_size 1`
- `--group_by_length true`
- `--packing false`
- `--padding_free false`
- 16 卡普通数据并行，先做 20–50 step 多卡 smoke，再决定完整 epoch

`delete` 会整条删除超过 max_length 的样本，不会安全地替用户完成轨迹截断，因此必须在训练前物理分桶。当前 NPU Transformers 路径没有可用 flash attention，packing/padding_free 不作为默认配置。

### 2. 课程和配比

建议分阶段，而不是一次混合所有长度：

1. `core_8k`：只用复判通过的 ≤8K 样本，验证工具协议、loss、评测和多卡稳定性。
2. `extension_16k`：加入复判通过的 8K–16K 样本。首个配比建议按样本 70% core / 30% extension；按当前平均长度换算，实际 token 预算约为 46%/54%，避免 extension 在算力上压倒核心样本。
3. `long_32k_review`：当前不进入正式 SFT。基础设施闭环后，可从 50% core / 40% extension / 10% long 的样本配比开始，长样本约占 27% token 预算，再根据 held-out 结果调整。

这些配比是首轮控制变量，不是最终最优值。应以 task-level test 上的工具闭环率、答案正确率、Hybrid 能力和错误恢复率决定是否增加长样本，而不是只看训练 loss。

### 3. 轨迹监督定义

轨迹 SFT 在优化器层面仍是 causal LM SFT，区别在于保留完整多轮 agent 轨迹并正确设置监督边界：

- assistant reasoning/final answer 和 tool call 参与 loss；
- tool response 保留在上下文，但不参与 loss；
- 工具 schema 必须进入 system；
- 错误恢复轨迹只有在恢复成功、最终答案正确时才作为正样本；
- 不把 partial/incorrect/incomplete 混进正向 SFT。

## 评测和验收

正式训练至少保留以下 task-level held-out 指标：

- JSON/tool schema 合法率；
- tool call 与 tool response 闭环率；
- 最终答案正确率；
- SQL/数据库结果一致率；
- KB 文档支持率；
- Hybrid 综合结论正确率；
- 工具错误后恢复成功率；
- 平均 tool call 数、重复调用率和输出 token；
- 8K、16K 分桶表现及不同来源模型分层表现。

训练 loss 只能说明拟合程度，不能替代以上 agent 行为评测。

## 局限性与风险

1. 显存结果来自单条样本、单个 optimizer step；正式 16 卡训练前仍需 20–50 step smoke 覆盖不同长度和消息形状。
2. 工具错误、网络、路径和凭据检测使用启发式规则，存在误报和漏报；它们用于隔离/排序，不替代人工或执行复判。
3. 现有 verdict 是复判起点，不是 gold。规划池中的 4,179 条弱证据样本不能在复判前物化为正式 SFT。
4. 当前规划只生成元数据索引，没有复制或提交轨迹正文，避免在质量结论未定时制造另一个“成品”副本。
5. 16K 安全方案明显偏 KB，Hybrid 数据不足；首轮结果不能外推为完整轨迹能力。
6. FSDP2 配置修复了层类名，但 32K 实测仍 OOM；不能因为配置文件存在就宣称 32K FSDP 可用。

## 后续优先级

1. 先复判 893 条 `core_8k`，建立可训练的第一批正样本和 held-out 评测。
2. 复判并路径归一化 `extension_16k`，完成 16 卡 20–50 step smoke。
3. 建立“错误后已恢复/未恢复”状态机，替代纯正则错误计数。
4. 对长 Hybrid 做语义压缩或补采短 Hybrid，避免 16K 数据类型失衡。
5. 只有在 SP/DeepSpeed/FSDP 任一路径达到 32K 峰值低于约 52 GiB 且连续训练稳定后，才启用长序列阶段。

## 回滚方式

本轮未修改原始 24 个 JSONL、manifest 或 verdict 文件。需要回滚时：
