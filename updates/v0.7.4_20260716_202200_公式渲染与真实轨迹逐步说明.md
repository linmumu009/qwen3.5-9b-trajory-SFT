# 公式渲染、labels 构造与真实轨迹逐步说明

## 一、更新概述

修订 `docs/trajectory_sft_process.md` 的三个讲解缺口：

1. 将不稳定的 `\[...\]`、`\(...\)` LaTeX 分隔符统一替换为 GitHub/Codex Markdown 支持更稳定的 `$$...$$`、`$...$`；
2. 重写 loss mask / labels 章节，说明它读取哪些结构信息、逐 token 输出什么、哪些位置变成 `-100`，以及 loss mask 与 causal/padding mask 的区别；
3. 在“从 JSON 到一次参数更新”的九个步骤中，把真实样本 `task_000201` 逐步带入，说明每一步的输入、实际动作和产物。

本轮只修改讲解材料，不改变训练数据、转换脚本、训练配置、模型权重或服务器产物。

## 二、具体改动

### 1. 公式渲染语法

- 21 个显示公式块全部使用独占行 `$$` 分隔；
- 所有行内变量统一使用 `$...$`；
- 清除文档中的 `\[`、`\]`、`\(`、`\)` 旧分隔符；
- 保留 softmax、masked NLL、batch loss、LoRA 和实际样本 loss 的数学含义。

例如 logits 定义现在写为：

$$
z_t=f_\theta(x_{\le t})\in\mathbb{R}^{|V|}
$$

### 2. loss mask / labels 输入与输出

新增完整的数据流：

```text
ms-swift agent row
  → Qwen3.5 template.encode(row)
  → input_ids + labels
  → shifted logits/labels
  → ignore labels == -100
  → masked cross-entropy
```

明确 labels 构造读取：

- `messages[i].role`；
- chat template 的消息和特殊 token 边界；
- `loss_scale="default+ignore_empty_think"`；
- 转换记录中的显式 `loss` 属性；
- batch padding 区域。

同时明确它不读取或判断 SQL 正确性；正确性筛选发生在编码之前。

新增逐位置定义：

$$
y_t=\begin{cases}
x_t,&m_t=1\\
-100,&m_t=0
\end{cases}
$$

并说明 `-100` 只取消该位置的直接交叉熵，不删除 `input_ids`，也不禁止后续 assistant 读取过去的 observation。

### 3. `task_000201` 的精确 mask 对照

逐段标注实际轨迹：

- tools/system/user → 保留在 `input_ids`，labels 为 `-100`；
- assistant 推理和 4 次 tool call → 保留真实 token id，参与 loss；
- 4 次 tool response → 保留在上下文，labels 为 `-100`；
- 查询返回 `6639` → 不作为直接 target；
- 最终回答“6639 分钟” → assistant target，参与 loss。

补充真实计数：

| 指标 | 数值 |
|---|---:|
| `input_ids` | 3,464 |
| `labels != -100` | 416 |
| `labels == -100` | 3,048 |
| 监督密度 | 12.0092% |

### 4. 九步训练流程逐项代入

在每个步骤增加“`task_000201` 实际发生的事”和本步产物：

1. 检查 4 次 tool call / response 闭环；
2. 核对 schema、聚合 SQL、工具返回和最终答案；
3. 映射 `<think>`、tool_call、tool_response 和工具 schema；
4. 得到 3,464 个真实 Qwen3.5 token，不发生截断；
5. 构造 causal、padding、loss 三种不同 mask；
6. 分别说明 schema 调用、SUM SQL 和最终回答阶段能看到什么；
7. 只对 416 个 supervised token 计算交叉熵；
8. 说明该样本如何贡献 batch 梯度，以及 observation 的间接梯度路径；
9. 说明全参数与 LoRA 只改变可训练参数集合，不改变本样本 labels 和 loss。

特别补充：不能把一条轨迹写成“一定单独执行一次 optimizer step”，因为实际更新可能聚合多个样本、多个 data-parallel rank 和多个 gradient-accumulation micro-batch。
